# Codex Adversarial Review

在写代码之前，先用对抗式思维审查方案。目标是在实现前发现问题，而不是部署后擦屁股。

## 输入

用户会描述一个要实现的功能或改动。如果没有描述，先问清楚再继续。

## 审查流程

### Phase 1: 理解范围

1. 列出这个改动会涉及的所有文件（读代码确认，不要猜）
2. 画出数据流：数据从哪来 → 经过什么处理 → 到哪去
3. 明确边界：哪些是要改的，哪些不能动

### Phase 2: 挑刺（核心）

从以下角度逐一攻击方案，每个角度至少给出一个具体质疑：

| 角度 | 要问的问题 |
|------|-----------|
| **状态一致性** | 这个改动会不会导致数据不一致？并发写入安全吗？SQLite WAL够用吗？ |
| **边界条件** | 空数据、极端值、超时、引擎崩溃时会怎样？ |
| **类型安全** | 有没有隐式any？字段访问会不会undefined？ |
| **时间/单位** | 涉及时间的地方，秒和毫秒分清楚了吗？ |
| **缓存陷阱** | Next.js的fetch缓存、静态生成、动态路由标记对吗？ |
| **向后兼容** | 老数据能兼容吗？需要迁移吗？迁移幂等吗？ |
| **性能** | N+1查询？大列表没分页？不必要的全表扫描？ |
| **安全** | SQL注入？XSS？权限检查遗漏？ |

### Phase 3: 替代方案

- 提出至少一个完全不同的实现路径
- 对比两种方案的优劣
- 如果替代方案明显更好，直说

### Phase 4: 结论

## 规则

- 不要当好好先生。如果方案有问题，直接说，不要用"也许可以考虑"这种软话
- 每个质疑必须具体到文件和代码行，不要泛泛而谈
- 如果方案本身就很好，也要说清楚为什么好，不要为了挑刺而硬挑
- 审查完才能开始写代码。如果有阻塞项，必须先解决
- 涉及3个以上文件的改动，必须先跑这个审查
